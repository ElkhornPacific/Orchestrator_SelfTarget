# Architecture: Docker for Orchestrator projects

This document defines the architecture elements for the docker initiative. Each ARC implements one or more requirements from the PRD; tasks reference these ARCs for drift coverage. System of record remains the Orchestrator and target repos; Docker is Processing/Compute only and SHALL NOT be SOR for PRD, Architecture, Tasks, GuardContract, or workflow-state.

ARC-001: Evaluation and recommendation document. Implements REQ-001, REQ-002, REQ-004. The initiative SHALL produce a documented evaluation (pros, cons, scope options, tool volatility, exit cost) and a clear recommendation (adopt / pilot / defer) with rationale. The document SHALL live in the initiative folder or docs/ and SHALL be auditable. If the recommendation is defer, the document SHALL include the rationale and conditions under which to revisit. PRD open questions (scope, pilot vs full adoption, image versioning) SHALL be resolved or carried forward in this document.

ARC-002: Optional Docker execution path. Implements REQ-003, REQ-005. If the recommendation is adopt or pilot, the initiative SHALL define the chosen scope (e.g. verification-only or CI-only) and SHALL produce or reference artifacts (Dockerfile, run instructions) sufficient to run the in-scope flow in a container. Docker SHALL remain optional for local development; the "run without Docker" path SHALL remain valid and documented. Gate semantics (PASS/FAIL, run-report) SHALL be unchanged when running in a container. Exit cost SHALL be documented.

ARC-003: Verification and documentation consistency. Implements coverage for all REQs and NFRs. The initiative SHALL verify that the evaluation document exists and contains recommendation and rationale; that if adopt/pilot, at least one supported container path exists and "run without Docker" remains valid; that there is no regression in workflow determinism or gate semantics; and that documentation makes it obvious whether Docker is required, optional, or CI-only.
