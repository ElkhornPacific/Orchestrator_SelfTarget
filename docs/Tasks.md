# Tasks: Docker for Orchestrator projects

Tasks are ordered for execution. Each task references one or more ARC IDs for drift coverage. TASK-002 is conditional on the recommendation from TASK-001 (adopt or pilot); if the recommendation is defer, TASK-002 is satisfied by ensuring the evaluation document from TASK-001 contains the deferral rationale and revisit conditions.

TASK-001: Produce the evaluation and recommendation document. Assess pros and cons of using Docker for Orchestrator projects; document scope options (verification-only, Orchestrator run only, CI only, full containerization); document tool volatility and exit cost (low/medium/high); state the recommended outcome (adopt / pilot / defer) with rationale. Resolve or state resolution path for PRD open questions (scope, pilot vs full adoption, image versioning). Output: a single document in the initiative folder (e.g. docs/initiatives/docker/evaluation-and-recommendation.md) or in docs/. If the recommendation is defer, include the rationale and conditions under which to revisit. References ARC-001.

TASK-002: If the recommendation from TASK-001 is adopt or pilot: define the chosen scope (e.g. verification-only: pytest and orchestrator.py in a container with repo bind-mount). Produce (1) a Dockerfile (or equivalent) that builds an image capable of running the in-scope flow, and (2) run instructions (one or two commands) in a discoverable place (README, docs/workflow.md, or initiative folder). Ensure the container run produces the same gate semantics (PASS/FAIL, run-report) as the non-Docker path. Document exit cost in or next to the evaluation document. Optionally add a short "Docker (optional)" subsection to docs/workflow.md or the initiative folder so operators see that Docker is optional or CI-only. If the recommendation is defer, no implementation; ensure the evaluation document from TASK-001 contains the deferral rationale and revisit conditions. References ARC-002.

TASK-003: Verify the evaluation document exists and contains recommendation and rationale; if adopt/pilot, verify at least one supported path to run the in-scope flow in a container and that "run without Docker" remains valid; verify no regression in workflow determinism or gate semantics; verify REQ-001 through REQ-005 are satisfied; verify documentation makes it obvious whether Docker is required, optional, or CI-only. References ARC-003.
